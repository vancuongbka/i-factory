## Description

A Bill of Materials (BOM) header defines what a factory produces and in what quantity. It records the product being manufactured (by name and optionally by a link to the product catalogue), the version of the recipe, the expected output quantity and unit, and whether this BOM is a phantom used only for costing purposes. The BOM header is the container that holds all the component line items and the full version history.

## Acceptance Criteria

- [ ] A BOM can be created with: `factoryId` (UUID, required), `code` (2â€“50 chars, required), `productName` (1â€“200 chars, required), `version` (max 20 chars, default `'1.0'`), `outputQuantity` (positive number, required), `outputUnit` (max 20 chars, required), `isPhantom` (boolean, default `false`), `notes` (max 1000 chars, optional), `productId` (UUID, optional), and at least one item in the `items` array.
- [ ] `productId` is optional â€” a BOM can exist without being linked to a product catalogue entry.
- [ ] When a linked product is deleted from the catalogue, `productId` is set to `NULL` on the BOM (`ON DELETE SET NULL`); the BOM itself is not deleted.
- [ ] Creating and updating a BOM header requires role `SUPER_ADMIN`, `FACTORY_ADMIN`, or `PRODUCTION_MANAGER`.
- [ ] Soft-deleting a BOM requires role `SUPER_ADMIN` or `FACTORY_ADMIN`.
- [ ] The list endpoint returns only BOMs where `isActive = true` for the factory, ordered by `code` ascending, with `items` relation loaded.
- [ ] The detail endpoint returns a single BOM (regardless of `isActive` status) with `items` relation loaded.
- [ ] A 404 `NotFoundException` is thrown when a BOM is not found within the factory scope.
- [ ] Updating a BOM header via `PATCH /:id` does not modify its items â€” the `updateBomV2Schema` explicitly omits the `items` field.
- [ ] Soft-delete sets `deletedAt`; hard deletion is not exposed.
- [ ] After creation, the API re-fetches and returns the full BOM (including persisted items) rather than returning the partially constructed object.
- [ ] The frontend create form auto-populates `productName` when a product is selected from the optional product dropdown.
- [ ] The frontend create form requires exactly one initial item (type: material only) to satisfy the schema's `items.min(1)` constraint.
- [ ] All mutation endpoints validate against `createBomV2Schema` / `updateBomV2Schema` via `ZodValidationPipe`.

## Inferred Business Rules

- **V2 schema is active; legacy schema retained for backward compatibility**: the controller uses `createBomV2Schema` (which supports `productId`, `isPhantom`, and multi-level item references). The original `createBomSchema` is exported from `api-types` but not used by any controller â€” it is labelled "legacy (kept for backward compat)".
- **productName is always required even when productId is provided**: `productName` is a free-text denormalised field. If linked to a product, the frontend auto-fills it from the product name, but the API does not enforce consistency between `productName` and the linked product's name.
- **isPhantom semantics**: phantom BOMs are used in costing calculations to represent intermediate assemblies that are not physically produced as discrete items. No special business logic enforces this flag; it is informational for downstream costing integrations.
- **isActive controls list visibility**: setting `isActive = false` via PATCH effectively archives a BOM from list views and dropdowns. Deactivated BOMs can still be fetched by ID.
- **Factory isolation**: all BOM queries are scoped to `factoryId` derived from the route parameter, enforced by `FactoryAccessGuard`.
- **Code is not enforced as unique**: no unique constraint on `(factoryId, code)` was observed in the entity. Multiple BOMs with the same code can coexist (differentiated only by `version` or `id`).

## Open Questions

1. **Code + version uniqueness**: the ERP sync conflict columns for `boms` are `[factoryId, code]` (without `version`), while routings use `[factoryId, code, version]`. Should `(factoryId, code, version)` be the unique key for BOMs as well, or is `(factoryId, code)` sufficient?
2. **productName/productId drift**: if the linked product's name is changed after the BOM is created, `productName` on the BOM is not updated. Should `productName` be a computed/derived field, or is denormalisation intentional for historical accuracy?
3. **Phantom BOM usage**: `isPhantom` is stored but no costing module references it. When will phantom BOM costing logic be implemented?
4. **BOM soft-delete and dependent entities**: production orders reference a `bomId`. When a BOM is soft-deleted, existing production orders retain the stale `bomId`. Should deletion be blocked if active production orders reference the BOM?
5. **outputUnit as free text**: `outputUnit` is a plain string (e.g., "PCS", "KG"), not a FK reference to the UOM catalogue. Changes to UOM names are not reflected. Is this intentional?
6. **Pagination on list**: no `page`/`limit` params are supported. Large BOM catalogues will return unbounded results.
7. **Inactive BOM in production orders**: can a deactivated BOM be assigned to a new production order? No guard prevents this at the API level.
8. **Legacy schema removal**: `createBomSchema` and `bomItemSchema` are still exported from `api-types`. When will they be removed to avoid confusion?
