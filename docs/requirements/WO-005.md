## Description

Each step within a work order can be individually marked as complete, recording a timestamp. The aggregate completion state of all steps drives the progress bar on the detail view and the step-count indicator on the list view, giving supervisors real-time visibility into how far through execution a work order is.

## Acceptance Criteria

- [ ] Each `WorkOrderStep` has an `isCompleted` boolean field (default `false`).
- [ ] Each `WorkOrderStep` has a `completedAt` nullable timestamp that records when the step was completed.
- [ ] Frontend detail page shows a progress bar derived from `(completedSteps / totalSteps) * 100`.
- [ ] Frontend list table shows a step completion ratio string (e.g. "2/5 steps completed") per work order.
- [ ] Frontend detail page steps table shows each step's status as "Completed" (green checkmark) or "Pending".
- [ ] Steps are displayed in ascending `stepNumber` order.
- [ ] Each step row shows: `stepNumber`, `name`, `workCenterId`, `estimatedMinutes`, `requiredSkills`, completion status.

## Inferred Business Rules

- `isCompleted` and `completedAt` are independent fields; setting `completedAt` does not automatically set `isCompleted = true` and vice versa. The service or caller is responsible for setting both consistently.
- Steps are cascade-deleted when their parent work order is deleted (hard or soft delete of the work order).
- Deleting a step individually is not supported via a dedicated endpoint; steps can only be managed as part of work order creation.
- `estimatedMinutes` captures total step time budget (setup + cycle). Actual time spent on a step is not tracked in the current schema (no `actualMinutes` field).
- `requiredSkills` is stored as a PostgreSQL `TEXT[]` array; the skill values are free-form strings, not foreign keys to the worker skill catalogue.
- `workCenterId` is nullable (SET NULL on work center deletion); a step can exist without a work center assignment.

## Open Questions

1. **Step completion API** — No `PATCH /work-orders/:id/steps/:stepId` endpoint was found. How does a worker mark a step as complete? Is step completion only possible via a full work order update, or is this endpoint missing?
2. **Automatic work order completion** — When all steps are marked complete, should the work order status automatically transition to `COMPLETED`? No such trigger was observed.
3. **Step re-opening** — Can a completed step (`isCompleted = true`) be un-completed? No business rule prevents setting `isCompleted` back to `false`.
4. **`requiredSkills` vs. skill catalogue** — Skills on steps are free-text strings, while the skill catalogue (MDM-006) uses structured `SkillEntity` records. There is no FK linkage. Is this intentional (loose coupling) or a gap?
5. **`actualMinutes` tracking** — Only `estimatedMinutes` exists; actual time on a step is not recorded. Is time tracking planned?
6. **Step ordering enforcement** — `stepNumber` is stored but there is no constraint preventing duplicate step numbers within the same work order.
