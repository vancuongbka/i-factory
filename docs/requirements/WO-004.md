## Description

A work order moves through a defined set of statuses from creation to completion or cancellation, reflecting where it is in the shop-floor execution cycle. A work order can optionally be assigned to a specific operator, enabling workload visibility and accountability.

## Acceptance Criteria

- [ ] Work order `status` is one of seven values: `PENDING`, `ASSIGNED`, `IN_PROGRESS`, `ON_HOLD`, `COMPLETED`, `REJECTED`, `CANCELLED`.
- [ ] A newly created work order always starts with `status = PENDING`.
- [ ] The `updateWorkOrderSchema` (partial) allows updating `status` and `assignedTo` via `PATCH` (schema exists; endpoint not yet confirmed in controller).
- [ ] `assignedTo` stores a UUID reference to a user; it is nullable (unassigned work orders are valid).
- [ ] `actualStartDate` and `actualEndDate` are nullable timestamps used to record real execution times (distinct from planned dates).
- [ ] Frontend status badges are colour-coded to distinguish active (`IN_PROGRESS`), blocked (`ON_HOLD`, `REJECTED`), and terminal (`COMPLETED`, `CANCELLED`) states.

## Inferred Business Rules

- Status transitions form an implicit state machine:
  - `PENDING` → `ASSIGNED` (operator assigned)
  - `ASSIGNED` → `IN_PROGRESS` (work begun)
  - `IN_PROGRESS` → `ON_HOLD` (paused)
  - `ON_HOLD` → `IN_PROGRESS` (resumed)
  - `IN_PROGRESS` → `COMPLETED` (all steps done, QC passed)
  - `IN_PROGRESS` → `REJECTED` (QC failed)
  - Any non-terminal status → `CANCELLED`
- The code does not enforce transition guards server-side; any valid enum value can be written via the update schema. Business-rule enforcement is implicit (not coded).
- `REJECTED` maps to a QC-fail outcome — a work order is rejected when the linked QC inspection fails (REQ-040 / REQ-042), though the link between QC result and work order status is not automated in the current code.
- Setting `status = ASSIGNED` without providing `assignedTo` is not prevented by the schema; the two fields are independent.
- `actualStartDate` / `actualEndDate` are not set automatically by the service on status transitions; they must be written explicitly by the caller.

## Open Questions

1. **Status transition enforcement** — No server-side guard checks that a status change follows the intended state machine. Is it acceptable to transition from `COMPLETED` back to `IN_PROGRESS`? Should a guard be added?
2. **Automatic `actualStartDate` on `IN_PROGRESS`** — Should the backend auto-set `actualStartDate = now()` when status changes to `IN_PROGRESS`? Similarly, auto-set `actualEndDate` on `COMPLETED` or `REJECTED`?
3. **`ASSIGNED` ↔ `assignedTo` coupling** — Should setting `assignedTo` automatically move status from `PENDING` to `ASSIGNED`, and clearing it move back to `PENDING`?
4. **PATCH endpoint** — `updateWorkOrderSchema` exists in `@i-factory/api-types` but no `PATCH /work-orders/:id` endpoint is visible in the controller. Is status/assignment update currently possible via the API?
5. **Role restrictions on status changes** — Can a `WORKER` or `OPERATOR` role change status (e.g., mark `IN_PROGRESS`), or is that also restricted to manager roles?
6. **QC → Work Order status link** — `REJECTED` implies a QC failure. Is there an automated side-effect that sets work order status to `REJECTED` when a linked QC inspection fails? No such coupling was found.
