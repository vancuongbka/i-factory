## Description

When a Bill of Materials is modified, engineers can create a formal revision to snapshot the current state before applying further changes. Each revision freezes the BOM header and all its items into an immutable record, records who created the revision and why (via optional change notes), and automatically increments the BOM version number. This provides a full audit trail of how a product's recipe evolved over time without losing the live, editable BOM.

## Acceptance Criteria

- [ ] Creating a revision requires role `SUPER_ADMIN`, `FACTORY_ADMIN`, or `PRODUCTION_MANAGER`.
- [ ] The `POST /:id/revise` endpoint accepts an optional `changeNotes` string (max 1000 chars).
- [ ] On revision creation, a `bom_revisions` record is saved with: `bomId`, `factoryId`, `fromVersion` (current BOM version), `toVersion` (bumped version), `revisedBy` (JWT `sub` claim of the requesting user), `changeNotes` (optional), and `snapshotData` (full BOM object including items as JSONB).
- [ ] The BOM's `version` field is updated to `toVersion` after the revision record is saved.
- [ ] Version bumping increments the last numeric segment of the version string: `"1.0"` → `"1.1"`, `"2.3"` → `"2.4"`, `"1.0.5"` → `"1.0.6"`.
- [ ] The `GET /:id/revisions` endpoint returns all revisions for the BOM scoped to `factoryId`, ordered by `createdAt` descending (newest first).
- [ ] The frontend revision log displays revisions in reverse chronological order (newest at the top).
- [ ] Each revision entry in the frontend shows: version transition (`fromVersion → toVersion`), creation timestamp (locale-formatted), and `changeNotes` when present.
- [ ] Attempting to revise a non-existent or out-of-scope BOM returns 404.
- [ ] Revision records are hard-deleted when the parent BOM is deleted (`CASCADE` on `bomId` FK).
- [ ] `revisedBy` stores the user UUID but the revision response does not resolve it to a display name.
- [ ] The `snapshotData` column is `JSONB` and stores an arbitrary object — its shape is not validated by a Zod schema at write time.

## Inferred Business Rules

- **Revisions are append-only**: there is no endpoint to update or delete an individual revision record. The audit trail is immutable by design.
- **Revision != rollback**: the system stores the snapshot but provides no endpoint to restore a BOM to a previous revision's state. Revisions are read-only historical records.
- **Version format is free text**: `version` is stored as a string (max 20 chars). The `bumpVersion` algorithm splits on `.` and increments the last segment. Non-numeric last segments (e.g., `"1.0-beta"`) would produce `NaN` via `parseInt`, resulting in a corrupted version string (`"1.NaN"`).
- **Revision is decoupled from item edits**: a revision is a manual action triggered explicitly by the user. There is no automatic revision on every save. Engineers must remember to create a revision before making breaking changes.
- **snapshotData includes `items` at revision time**: `{ ...bom, items: bom.items }` is saved, capturing the full item list. However, `items` is loaded from the `items` relation of the BomEntity — if the relation is not eager-loaded at call time, `items` may be `undefined` in the snapshot.
- **factoryId is stored on each revision**: redundant with the parent BOM's `factoryId`, but allows querying revisions directly by factory without joining to the BOM table.

## Open Questions

1. **Non-numeric version segment**: `bumpVersion("1.0-beta")` splits to `["1", "0-beta"]`, then `parseInt("0-beta")` returns `0`, yielding `"1.1"`. This is probably acceptable but untested. What is the allowed version format? Should a regex validator be added to the schema?
2. **Snapshot completeness**: `snapshotData` is set to `{ ...bom, items: bom.items }` which relies on TypeORM lazy relation loading. If `bom.items` is not populated (e.g., the BOM was fetched without the `relations: ['items']` option), the snapshot will record `items: undefined`. Should a fresh `findById` (which does load items) be used as the snapshot source?
3. **Rollback / restore endpoint**: there is no way to restore a BOM to a past revision state. Is a "restore from revision" feature planned?
4. **Revision on soft-delete**: when a BOM is soft-deleted, no automatic revision is created. The last state before deletion is not preserved in a revision. Should soft-delete trigger an automatic revision?
5. **Version after revision on update**: the `PATCH /:id` header update endpoint does not trigger a version bump or revision. Is this intentional? Users can modify the BOM header silently without any version change.
6. **Revision list pagination**: `findRevisions` has no `LIMIT` — a BOM with many revisions returns all records in one query. Should pagination be added?
7. **`revisedBy` user resolution**: the revision record stores a UUID. The frontend revision log does not display the user's name, only the timestamp and version. Should a join or user-name lookup be added to the revision list response?
8. **Concurrent revision creation**: if two users trigger a revision simultaneously, both reads of `bom.version` return the same value, both write the same `fromVersion`, and one will silently overwrite the other's version bump. A database-level lock or optimistic concurrency check is absent.

## Context for Claude

- **Stack**: NestJS, TypeORM, PostgreSQL (JSONB column for snapshot), Zod, Next.js 15 App Router with TanStack Query.
- **Guard chain**: `JwtAuthGuard → RolesGuard → FactoryAccessGuard`. The `@CurrentUser()` decorator extracts the JWT `sub` claim as `revisedBy`.
- **Entity**: `bom_revisions` table. FK `bomId → boms.id` with `CASCADE`. No FK on `revisedBy` (UUID only, no join to `users`). `snapshotData` is a raw JSONB column.
- **Version bump logic**: `bumpVersion()` is a private method in `BomService` — splits on `.`, increments last integer segment.
- **Hooks**: `useCreateBomRevision(bomId)`, `useBomRevisions(bomId)` in `apps/web/src/hooks/use-bom.ts`.
- **Related features**: BOM-001 (header owns `version`), BOM-002 (items captured in snapshot).
