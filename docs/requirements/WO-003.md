## Description

When a manufacturing routing exists for the product being produced, a production manager can create a work order and have its steps automatically populated from that routing's operations. This removes the need for manual step entry and ensures the floor-level execution sequence matches the engineered process definition.

## Acceptance Criteria

- [ ] `POST /factories/:factoryId/work-orders/:productionOrderId/from-routing` creates a work order and auto-generates steps.
- [ ] The service looks up the production order by `:productionOrderId` and retrieves its linked `productId`.
- [ ] The service queries for the **active** routing for that product (routing with `isActive = true`), including its operations.
- [ ] Each routing operation is mapped to a work order step: `sequence → stepNumber`, `name → name`, `(cycleTime + setupTime) → estimatedMinutes`, `requiredSkills → requiredSkills`, `workCenterId → workCenterId`.
- [ ] If no active routing is found, the service **falls back** to the `steps[]` array supplied in the request body (same as manual creation).
- [ ] If neither a routing nor any DTO steps are provided, the service throws a validation error (no empty-step work orders via this endpoint).
- [ ] The created work order starts with `status = PENDING`.
- [ ] Access is restricted to `SUPER_ADMIN`, `FACTORY_ADMIN`, or `PRODUCTION_MANAGER`.
- [ ] Frontend form provides a "From Routing" mode toggle that hides the manual step rows.
- [ ] TanStack Query cache for work orders is invalidated after a successful create.

## Inferred Business Rules

- Only the **active** routing version (`isActive = true`) is used; if multiple versions exist for a product, inactive versions are ignored.
- The fallback to DTO steps is a safety net — callers should not rely on it when a routing is expected. No warning is surfaced to the caller when fallback occurs.
- Routing operations define the canonical step order via `sequence`; this maps 1-to-1 to `stepNumber`.
- `estimatedMinutes` on the generated step combines both `cycleTime` and `setupTime` from the routing operation into a single value (cycle + setup).
- The `:productionOrderId` path segment creates a potential ambiguity with the generic `GET /:id` route on the same controller. This is noted as a known risk (see Open Questions in `_index.md`, item 9).

## Open Questions

1. **Fallback transparency** — When the routing fallback occurs, no warning is returned to the caller. Should the API response indicate whether steps were generated from routing or from the DTO?
2. **Multiple active routings** — The code assumes at most one active routing per product. If more than one routing has `isActive = true`, which one is selected (first found)? Should a uniqueness constraint prevent this?
3. **Routing snapshot vs. live link** — Steps are created as independent rows; they are not linked back to the source routing or operation. If the routing is later changed, existing work order steps are unaffected. Is this the intended behaviour (snapshot semantics)?
4. **Route collision** — `POST /:productionOrderId/from-routing` and `GET /:id` share the same path prefix. Does the NestJS router correctly distinguish them given both use a UUID-shaped segment?
5. **Cross-factory routing lookup** — The routing lookup uses `productId` but does not explicitly filter by `factoryId`. Could a routing from a different factory be selected if product IDs overlap?
