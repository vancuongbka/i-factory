## Description

Each Bill of Materials is composed of one or more line items (components) that describe what raw materials or sub-assemblies are needed to manufacture the finished product. Each item specifies a quantity, unit of measure, and an optional waste allowance. Items can reference either a raw material from the inventory catalogue or another BOM (enabling multi-level, hierarchical assemblies). The sequence of items within a BOM is tracked to preserve display and processing order.

## Acceptance Criteria

- [ ] A BOM must contain at least one item at creation time (`items.min(1)` enforced by `createBomV2Schema`).
- [ ] Each item must have exactly one of `materialId` (UUID) or `childBomId` (UUID); providing neither is rejected with validation error "Each BOM item must have either materialId or childBomId".
- [ ] Item `quantity` must be a positive number (> 0).
- [ ] Item `unit` must be a string with maximum 20 characters.
- [ ] Item `wastePercentage` must be a number between 0 and 100 inclusive; defaults to `0` when omitted.
- [ ] Item `sequence` must be a positive integer; defaults to `1` when omitted.
- [ ] Item `notes` is optional and must not exceed 500 characters.
- [ ] Adding, updating, and removing items requires role `SUPER_ADMIN`, `FACTORY_ADMIN`, or `PRODUCTION_MANAGER`.
- [ ] Adding an item to a non-existent or out-of-scope BOM returns 404.
- [ ] Removing an item that does not belong to the specified BOM returns 404.
- [ ] Items are stored with `CASCADE` delete — removing the parent BOM also removes all its items.
- [ ] Item removal uses hard delete (`itemRepo.delete()`), not soft delete.
- [ ] The BOM detail endpoint (`GET /:id`) always returns items loaded as a relation, ordered by the database default (insertion order / by `sequence` ascending on the frontend sort).
- [ ] When a child BOM referenced by an item is deleted, `childBomId` on the item is set to `NULL` (`ON DELETE SET NULL`) — the item record is preserved.
- [ ] The frontend item editor auto-calculates the next `sequence` as `max(existing sequences) + 1`; the first item defaults to sequence `1`.
- [ ] The frontend item editor displays items sorted by `sequence` ascending.

## Inferred Business Rules

- **Item type is mutually exclusive**: an item is either a "material" (points to `materialId`) or a "child BOM" (points to `childBomId`). The Zod `.refine()` enforces that at least one is present, but there is no schema-level constraint preventing *both* from being set simultaneously — the backend would accept a row with both set.
- **Hierarchical BOM support**: by allowing `childBomId`, a BOM can include another BOM as a component. This enables multi-level assembly structures. The `childBom` relation uses `SET NULL` on delete, so deleting a child BOM does not cascade-delete the item row in the parent.
- **Waste percentage is informational**: `wastePercentage` is stored but no backend logic automatically inflates the required quantity based on it. Waste factor application is assumed to be handled by downstream costing or MRP consumers.
- **Sequence gaps are allowed**: the auto-increment logic (`max + 1`) can produce gaps if items are removed. No re-sequencing occurs on removal.
- **Item update does not validate type exclusivity at schema level**: the `PATCH /:id/items/:itemId` route uses `Partial<AddBomItemDto>` without `UsePipes`, meaning Zod validation is skipped on updates. Partial updates are passed directly to `itemRepo.save()`.
- **No item reordering endpoint**: there is no dedicated endpoint to bulk-reorder items by sequence. Sequence can only be set at item creation or via the generic PATCH update.

## Open Questions

1. **Both `materialId` and `childBomId` set simultaneously**: the Zod refine only checks "at least one is set", not "exactly one". Should a constraint prevent setting both on the same item? The database has no unique exclusion constraint enforcing this.
2. **`PATCH` item validation gap**: `updateItem` uses `Partial<AddBomItemDto>` without `@UsePipes`, bypassing Zod validation. Is this intentional for partial updates, or should the same `addBomItemSchema.partial()` validation be applied?
3. **Material ID is a raw UUID on the frontend create form**: the initial item on the create form requires the user to type a raw UUID for `materialId`. No material lookup/search dropdown is provided. Is this a known UX gap or a placeholder?
4. **childBomId is also a raw UUID in the item editor**: same issue as above — users must type or paste a UUID to set a child BOM reference. Should a BOM selector dropdown be added?
5. **Item hard delete vs soft delete**: BOM items use hard delete (`DELETE` SQL) while the parent BOM uses soft delete (`deletedAt`). Is this asymmetry intentional? There is no audit trail for removed items.
6. **Circular BOM references**: nothing prevents BOM A referencing BOM B as a child item while BOM B also references BOM A. No cycle-detection logic exists.
7. **Item quantity precision**: `quantity` is stored as `DECIMAL(12,4)` in the DB but the Zod schema only requires `number().positive()` with no decimal-place constraint. Is 4 decimal places sufficient for all units?
8. **Waste percentage not applied**: `wastePercentage` is stored but unused in the codebase. Which module is expected to consume it (MRP, costing, production planning)?

## Context for Claude

- **Stack**: NestJS, TypeORM, PostgreSQL, Zod, Next.js 15 App Router with TanStack Query.
- **Guard chain**: `JwtAuthGuard → RolesGuard → FactoryAccessGuard` on all item endpoints.
- **Entity**: `bom_items` table. FK `bomId → boms.id` with `CASCADE` delete. FK `childBomId → boms.id` with `SET NULL`. FK `materialId` is an unresolved UUID — no FK constraint to `materials` table observed in the entity.
- **Hooks**: `useAddBomItem(bomId)`, `useRemoveBomItem(bomId)` in `apps/web/src/hooks/use-bom.ts`.
- **Related features**: BOM-001 (header), BOM-003 (revisions). Item snapshot is captured in `snapshotData` during revision creation.
- **Update route note**: `PATCH /factories/:factoryId/bom/:id/items/:itemId` does not use `@UsePipes` — validation is skipped.
