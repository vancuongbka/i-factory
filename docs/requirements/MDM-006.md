## Description

The skill catalogue defines the competencies that workers need to perform specific manufacturing operations. Skills are referenced in routing operations and work order steps to declare which worker qualifications are required to execute that step. This allows planners to match the right operators to the right tasks during work order assignment.

## Acceptance Criteria

- [ ] A skill can be created with: `factoryId` (UUID, required), `code` (2–50 chars, required), `name` (1–200 chars, required), `level` (required enum), `description` (max 500 chars, optional).
- [ ] `level` must be one of: `BASIC`, `INTERMEDIATE`, `ADVANCED`, `EXPERT`.
- [ ] Creating and updating a skill requires role `SUPER_ADMIN`, `FACTORY_ADMIN`, or `PRODUCTION_MANAGER`.
- [ ] Soft-deleting a skill requires role `SUPER_ADMIN` or `FACTORY_ADMIN`.
- [ ] The list endpoint returns only `isActive = true` skills for the factory, ordered by `code` ascending.
- [ ] A 404 `NotFoundException` is thrown when attempting to update or delete a skill not found within the factory scope.
- [ ] Soft-delete is used; hard deletion is not available.
- [ ] The frontend skills table shows: code, name, level badge, isActive status, and edit/delete actions.
- [ ] The `isActive` field is not present in the `createSkillSchema`; its initial value defaults to `true` at the database level.

## Inferred Business Rules

- **Factory isolation**: all skill operations are scoped to `factoryId`.
- **Skills stored as strings in routing operations**: routing operation's `requiredSkills` column is a `text[]` array, not a UUID array referencing the skills table. This means skills in routing operations are free-text labels, not FK references to the skill catalogue. Renaming or deleting a skill has no effect on existing routing operation skill requirements.
- **Skills stored as strings in work order steps**: similarly, `work_order_steps.requiredSkills` is a `text[]`. The catalogue serves as a vocabulary reference, not an enforced constraint.
- **isActive as list filter**: inactive skills are hidden from list results and therefore from dropdowns in routing/work order forms. Existing references in routing operations and steps retain their text values.
- **level is informational**: the `level` enum documents the proficiency required but no automated matching logic uses it to filter operator assignments.

## Open Questions

1. **Skill reference integrity**: `requiredSkills` in routing operations and work order steps is a `text[]` — free-form strings, not FK references. Should the system enforce that skill names in these arrays match existing, active skill records?
2. **Skill rename impact**: if a skill name is changed, existing routing operations that reference the old name as a string are not updated. Is there a plan to move to UUID-based skill references for consistency?
3. **Worker-to-skill assignment**: the current model defines skills as factory-level definitions but does not link skills to individual user/operator profiles. Is worker skill assignment (e.g., `user_skills` table) planned?
4. **Deletion protection**: since `requiredSkills` is a text array with no FK, deleting a skill silently leaves stale references in routing operations and work order steps. Should a check warn or block deletion when references exist?
5. **isActive in createSkillSchema**: the `createSkillSchema` does not include `isActive`. Is there a way to create a skill in an inactive state, or does it always start active?
6. **Code uniqueness**: no unique constraint on `(factoryId, code)` was found.

## Context for Claude

- **Stack**: NestJS, TypeORM, PostgreSQL, Zod, Next.js 15 App Router.
- **Entity**: `skills` table. No FK outbound. Referenced by `routing_operations.requiredSkills` (text[]) and `work_order_steps.requiredSkills` (text[]) — loose coupling by convention, not by constraint.
- **Related**: MDM-007 (Routings) uses skill names in operation `requiredSkills`. Work Orders module uses skill names in step `requiredSkills`.
